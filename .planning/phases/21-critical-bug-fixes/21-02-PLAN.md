**MBUG-05: Biometric logout awaits completion.**
In `AuthProvider.tsx` (around lines 51-57), the `.then()` callback after `authenticateAsync` calls `logout()` without `await`. Fix by making the `.then()` callback `async` and awaiting `logout()` before setting `setBiometricChecked(true)` and `setIsReady(true)`. The corrected flow:

```typescript
.then(async (result) => {
  if (!result.success) {
    await useAuthStore.getState().logout();
  }
  setBiometricChecked(true);
  setIsReady(true);
})
```

This ensures tokens are fully cleared before the navigation guard runs.

**MBUG-06: Return wizard hydration guard.**
In all three return wizard screens (`mileage.tsx`, `checklist.tsx`, `confirm.tsx`), the `useEffect` that redirects when `!rentalId` fires before Zustand `persist` hydration completes. Fix by:

1. Import or use the hydration state from the return draft store. Zustand persist exposes `useReturnDraftStore.persist.hasHydrated()` and an `onFinishHydration` callback. The simplest approach: add a `const hasHydrated = useReturnDraftStore.persist.hasHydrated();` check (or use `useSyncExternalStore` to subscribe to hydration). If the store has a `_hasHydrated` field, use that.

2. In the navigation guard useEffect, add `hasHydrated` to the dependency array and gate the redirect: `if (hasHydrated && !rentalId) { router.replace('/(tabs)/rentals'); }`. This prevents the redirect from firing on the initial render when the store is still loading from AsyncStorage.

Check the store definition to see if `_hasHydrated` is already exposed. If not, add a simple hydration hook:
```typescript
const useHasHydrated = () => {
  const [hydrated, setHydrated] = useState(useReturnDraftStore.persist.hasHydrated());
  useEffect(() => {
    const unsub = useReturnDraftStore.persist.onFinishHydration(() => setHydrated(true));
    return () => { unsub(); };
  }, []);
  return hydrated;
};
```

Place this hook in the store file or a shared hooks file. Use it in all three return screens.

**MBUG-07: ErrorBoundary retry increments key to force remount.**
The current ErrorBoundary likely resets its `hasError` state on retry but does not force children to remount. Children that threw may have corrupted internal state that persists across re-renders.

Fix: Add a `retryKey` counter to the ErrorBoundary state. In the retry handler, increment `retryKey` and clear `hasError`. In `render()`, wrap `this.props.children` with a `key={this.state.retryKey}` wrapper (a Fragment with key, or wrap children in a View with key). When the key changes, React unmounts and remounts the entire child tree, giving children a fresh start.

```typescript
state = { hasError: false, error: null, retryKey: 0 };

handleRetry = () => {
  this.setState((prev) => ({
    hasError: false,
    error: null,
    retryKey: prev.retryKey + 1,
  }));
};

render() {
  if (this.state.hasError) {
    return <ErrorFallback onRetry={this.handleRetry} error={this.state.error} />;
  }
  return <React.Fragment key={this.state.retryKey}>{this.props.children}</React.Fragment>;
}
```

If the component uses a different pattern (e.g., function component with error boundary hook), adapt accordingly but ensure the key-based remount is present.
