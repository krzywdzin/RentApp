1. The columns access `(row.original as any).vehicle` and `(row.original as any).customer`. This happens because the column definitions use `RentalDto` which doesn't have vehicle/customer relations.
2. Create an extended interface in columns.tsx or a local types file:
   ```typescript
   interface RentalWithRelations extends RentalDto {
     vehicle?: { registration: string; make: string; model: string };
     customer?: { firstName: string; lastName: string };
   }
   ```
3. Update the column definition to use `ColumnDef<RentalWithRelations>` instead of `ColumnDef<RentalDto>`.
4. Then access `row.original.vehicle?.registration` directly without `as any`.

**wynajmy/[id]/page.tsx — Remove `as any` casts (lines ~224, 229-230):**

1. Same pattern — the rental detail page accesses `(rental as any).vehicle` and `(rental as any).customer`.
2. Use the same `RentalWithRelations` interface (import from columns.tsx or define locally).
3. Cast the API response once at the point of use: `const rentalData = rental as RentalWithRelations;` then access `.vehicle?.registration` and `.customer?.firstName` cleanly.
4. Alternatively, if `useRental` returns the data, extend its generic type.

**wynajmy/nowy/page.tsx — Remove `zodResolver(formSchema) as any` (line ~67):**

1. The `as any` cast on zodResolver indicates a type mismatch between the Zod schema and react-hook-form's expected resolver type.

3. If type error persists, use a targeted type assertion: `zodResolver(formSchema) as Resolver<z.infer<typeof formSchema>>` with import of `Resolver` from `react-hook-form`. This is more specific than `any`.
